# ServeRest — Test Design & Coverage Analysis

Test design document for the three core flows under test: **User Registration**, **Login**,
and **Product Registration**. Every rule and status code below was verified directly against
the running ServeRest API (not assumed from the docs) — see the evidence notes inline. Every
API-level scenario identified in this analysis has been automated; UI-level scenarios were
selectively automated based on the candidate analysis in [Section 6](#6-ui-automation-candidate-analysis).

## Legend

| Symbol | Meaning                                                             |
| ------ | ------------------------------------------------------------------- |
| ✅     | Automated in this repository (see file referenced)                  |
| 📋     | Documented / recommended, not automated — see rationale in the row  |
| ⚠️     | Real gap or inconsistency found in the application while probing it |

## 1. Flow: User Registration (`POST /usuarios`, UI `/cadastrarusuarios`)

### Confirmed field rules

| Field           | Required | Validation observed                                                                                                                       | Boundary                                                                                          |
| --------------- | -------- | ----------------------------------------------------------------------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------- |
| `nome`          | yes      | any non-empty string                                                                                                                      | ⚠️ whitespace-only (`"   "`) is **accepted**; ⚠️ no max length — a 1000-char name is **accepted** |
| `email`         | yes      | must contain a valid e-mail shape (`"email deve ser um email válido"`); must be globally unique (`"Este email já está sendo usado"`, 400) | no observed max length                                                                            |
| `password`      | yes      | any non-empty string                                                                                                                      | no minimum length enforced (`"123"` is accepted)                                                  |
| `administrador` | yes      | must be the literal string `"true"` or `"false"` (`"administrador deve ser 'true' ou 'false'"`, 400)                                      | boolean-as-string, not a real boolean                                                             |

### Scenario matrix

| Dimension                | Scenario                                                                    | Expected                                                                                                            | Status                                   |
| ------------------------ | --------------------------------------------------------------------------- | ------------------------------------------------------------------------------------------------------------------- | ---------------------------------------- |
| Happy path               | Valid unique data, `administrador:"true"`                                   | `201` + `{ message, _id }`                                                                                          | ✅ `users.cy.js`                         |
| Happy path               | Valid unique data, `administrador:"false"`                                  | `201` + `{ message, _id }`                                                                                          | ✅ `users.cy.js`                         |
| Negative                 | Duplicate e-mail                                                            | `400` `"Este email já está sendo usado"`                                                                            | ✅ `users.cy.js`                         |
| Negative                 | All fields missing (`{}`)                                                   | `400`, one message per missing field                                                                                | ✅ `users.cy.js`                         |
| Negative                 | Invalid e-mail shape (no `@`)                                               | `400` `"email deve ser um email válido"`                                                                            | ✅ `users.cy.js`                         |
| Negative                 | `administrador` outside `"true"`/`"false"` (e.g. `"maybe"`)                 | `400` `"administrador deve ser 'true' ou 'false'"`                                                                  | ✅ `users.cy.js`                         |
| Equivalence partitioning | E-mail: valid shape (accepted) vs. invalid shape (rejected)                 | Valid → `201`; invalid → `400`                                                                                      | ✅ covered by the two tests above        |
| Boundary value           | `nome` length: 0 chars / 1000 chars                                         | 0 chars → `400` (covered by "missing fields"); 1000 chars → **accepted** ⚠️                                         | ✅ `users.cy.js`                         |
| Edge case                | `nome` = whitespace-only (`"   "`)                                          | Business expectation: rejected. **Actual: accepted (201)**                                                          | ✅ characterization test, ⚠️ real gap    |
| Edge case                | Leading/trailing whitespace in `email`                                      | Not probed — recommend testing, likely accepted verbatim (no trim)                                                  | 📋 low priority, no observed user impact |
| Invalid user behavior    | Submitting the registration form twice quickly (UI double-click)            | Should not create two accounts / should disable the button while submitting                                         | 📋 UI-level, see Section 6               |
| Implicit business rule   | Nothing prevents self-registering as `administrador:"true"` via the API     | Privilege escalation by design in this training app — a real finding in a production audit                          | ⚠️ noted, out of scope to "fix"          |
| API failure              | Malformed JSON body (unquoted keys/values)                                  | `400` with a guidance message                                                                                       | ✅ `users.cy.js`                         |
| Data validation          | `id` format on `GET /usuarios/:id`                                          | Must be exactly 16 alphanumeric chars, else `400`                                                                   | ✅ `users.cy.js`                         |
| Inconsistency ⚠️         | `DELETE /usuarios/:id` with a malformed id (wrong length)                   | Unlike `GET`, `DELETE` does **not** validate id shape — returns `200` `"Nenhum registro excluído"` instead of `400` | ✅ characterization test, ⚠️ real gap    |
| Permissions              | Registering as `administrador:"false"` and later trying to create a product | See Flow 3 → Permissions                                                                                            | ✅ `products.cy.js`                      |

## 2. Flow: Login (`POST /login`, UI `/login`)

### Confirmed rules

| Rule                                                                                               | Evidence                                                                                                          |
| -------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------- |
| Success → `200` + `{ message, authorization }`, token format `Bearer <JWT>`                        | ✅ verified                                                                                                       |
| Invalid credentials (wrong password or unregistered e-mail) → `401` `"Email e/ou senha inválidos"` | ✅ verified, same message for **both** cases (no user enumeration — good practice)                                |
| Missing `email` / `password` → `400` with a per-field message                                      | ✅ verified                                                                                                       |
| Token TTL                                                                                          | JWT `exp - iat = 600s` (**10 minutes**), no refresh-token mechanism observed ✅ verified by decoding a real token |

### Scenario matrix

| Dimension                   | Scenario                                                                                                    | Expected                                                                                                                                                                                      | Status                                                                                                                                                                                                                  |
| --------------------------- | ----------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| Happy path                  | Valid e-mail + password, regular user                                                                       | `200` + token; UI redirects to `/home`                                                                                                                                                        | ✅ `login.cy.js`, `login.cy.js` (UI)                                                                                                                                                                                    |
| Happy path                  | Valid e-mail + password, admin user                                                                         | `200` + token; UI redirects to `/admin/home` (different destination and heading than the regular-user case)                                                                                   | ✅ `login.cy.js` (UI)                                                                                                                                                                                                   |
| Negative                    | Wrong password for a real account                                                                           | `401` `"Email e/ou senha inválidos"`                                                                                                                                                          | ✅ `login.cy.js` (API + UI)                                                                                                                                                                                             |
| Negative                    | Non-existent e-mail                                                                                         | `401`, identical message (no account enumeration) — distinct equivalence partition from "wrong password"                                                                                      | ✅ `login.cy.js`                                                                                                                                                                                                        |
| Assertion quality ⚠️        | Asserting only `cy.url().should('include', '/home')` after login                                            | Looks sufficient, but `/admin/home` also contains the substring `/home`, and both navbars render the same `[data-testid="logout"]` — a test seeding an admin user by mistake would still pass | ⚠️ found while strengthening this exact test; fixed by asserting the page-specific `h1` heading (`"Serverest Store"` vs. `"Bem Vindo"`) in addition to the URL — now two dedicated tests, one per redirect target       |
| Negative                    | Missing `email` / missing `password`                                                                        | `400` with per-field message                                                                                                                                                                  | ✅ `login.cy.js`                                                                                                                                                                                                        |
| Edge case                   | Case-sensitivity of e-mail (`Test@qa.com` vs `test@qa.com`)                                                 | Not probed — recommend testing; a real source of "can't log in" support tickets if inconsistent with registration                                                                             | 📋 not automated                                                                                                                                                                                                        |
| Edge case                   | Leading/trailing whitespace in e-mail/password on submit                                                    | Not probed — UI does not visibly trim inputs                                                                                                                                                  | 📋 not automated                                                                                                                                                                                                        |
| Security / invalid behavior | Brute-force attempts (many rapid invalid logins)                                                            | No rate limiting/lockout observed in ServeRest — acceptable for a training app, would be a finding in a real audit                                                                            | ⚠️ noted                                                                                                                                                                                                                |
| Session expiration          | Token used after the 10-minute `exp`                                                                        | Expect `401` from `/produtos`/other protected routes                                                                                                                                          | 📋 **not automated by design**: the server checks real wall-clock time inside the JWT library, so `cy.clock()` cannot fake it; the only faithful test would sleep 10+ minutes, which is a poor trade-off for a CI suite |
| Session expiration          | UI behavior when the token expires mid-session (no visible session/idle warning)                            | Front does not appear to proactively warn before expiry                                                                                                                                       | 📋 UX gap to flag, see Section 6                                                                                                                                                                                        |
| Concurrency                 | Same user logs in from two tabs/devices simultaneously                                                      | ServeRest issues a new stateless JWT per login; both remain valid until each expires (no single-session enforcement)                                                                          | 📋 documented, low value to automate (expected/stateless-by-design behavior, not a bug)                                                                                                                                 |
| Network                     | Login request times out / connection drops mid-request                                                      | UI should show a network-error state, not hang indefinitely                                                                                                                                   | 📋 candidate — see Section 6 (`cy.intercept` + forced network error)                                                                                                                                                    |
| API failure                 | Login while the API is down (matches the `serverest.dev` 503 outage observed while working on this project) | Front should show a graceful error, not a blank/broken screen                                                                                                                                 | ⚠️ **actually observed** during this project: the public front shows no useful fallback UI when the backend 503s                                                                                                        |
| Localization                | Error/success messages are hardcoded in Portuguese (`pt-BR`) with no i18n layer                             | N/A for this app (single-locale by design)                                                                                                                                                    | 📋 documented, not a defect                                                                                                                                                                                             |

## 3. Flow: Product Registration (`POST /produtos`, UI `/cadastrarprodutos`, admin only)

### Confirmed field rules

| Field        | Required | Validation observed                                                                                               | Boundary                                                                                            |
| ------------ | -------- | ----------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------- |
| `nome`       | yes      | must be unique across all products (`"Já existe produto com esse nome"`, 400)                                     | not probed for length limits                                                                        |
| `preco`      | yes      | must be a **positive whole number** — `0`/negative rejected (`"...positivo"`); decimals rejected (`"...inteiro"`) | valid range is integers in `[1, ∞)` — discovered mid-session, initially assumed decimals were valid |
| `quantidade` | yes      | must be `>= 0` (`"quantidade deve ser maior ou igual a 0"`)                                                       | valid range is `[0, ∞)`, **inclusive** — inconsistent boundary semantics vs. `preco`                |
| `descricao`  | yes      | not deeply probed                                                                                                 | 📋                                                                                                  |

### Scenario matrix

| Dimension       | Scenario                                                                                             | Expected                                                                                                             | Status                                                   |
| --------------- | ---------------------------------------------------------------------------------------------------- | -------------------------------------------------------------------------------------------------------------------- | -------------------------------------------------------- |
| Happy path      | Valid unique product + valid admin token                                                             | `201` + `{ message, _id }`                                                                                           | ✅ `products.cy.js`                                      |
| Negative        | No `authorization` header                                                                            | `401` `"Token de acesso ausente, inválido, expirado ou usuário do token não existe mais"`                            | ✅ `products.cy.js`                                      |
| Permissions     | Valid token but **non-admin** user (`administrador:"false"`)                                         | `403` `"Rota exclusiva para administradores"` — **distinct status/message from the no-token case**                   | ✅ `products.cy.js`                                      |
| Negative        | Duplicate product name                                                                               | `400` `"Já existe produto com esse nome"`                                                                            | ✅ `products.cy.js`                                      |
| Boundary value  | `preco` at `0`, `-50`, `1` (smallest valid), `1.5` (non-integer)                                     | `0`/`-50` → `400` "positivo"; `1` → `201`; `1.5` → `400` "inteiro"                                                   | ✅ `products.cy.js`                                      |
| Boundary value  | `quantidade` at `0`, `-1`                                                                            | `0` → `201` (accepted, inclusive bound); `-1` → `400`                                                                | ✅ `products.cy.js`                                      |
| Edge case       | Garbage/invalid token (e.g. `Bearer garbage-token-123`)                                              | `401`, same generic message as "no token"                                                                            | ✅ `products.cy.js`                                      |
| Edge case       | Token with `authorization` header missing the `Bearer ` prefix                                       | `401`, same generic message                                                                                          | ✅ `products.cy.js`                                      |
| Data validation | Non-numeric `preco` (string)                                                                         | `400` `"preco deve ser um número"` (distinct message from the "must be positive"/"must be an integer" cases)         | ✅ `products.cy.js`                                      |
| Permissions     | `GET /produtos` without any token (read scope)                                                       | `200` — reads are public, only writes require admin                                                                  | ✅ `products.cy.js`                                      |
| Concurrency     | Two admins create a product with the same name in parallel (race)                                    | One should win with `201`, the other should get `400` duplicate — real TOCTOU race risk in a non-transactional store | 📋 not automated — see Section 6 rationale               |
| API failure     | Product creation while `/usuarios` (auth dependency) is unreachable                                  | Should fail gracefully with `401`/`5xx`, not crash the UI form                                                       | 📋 candidate — see Section 6                             |
| Timeout         | Slow network on submit (UI)                                                                          | Submit button should show a loading state and not allow duplicate submits                                            | 📋 — UI has no visible loading indicator, worth flagging |
| Accessibility   | Form fields have associated labels / are keyboard-navigable / errors are announced to screen readers | Not audited in this pass — recommend an `axe-core` (`cypress-axe`) pass over `/cadastrarprodutos` and `/login`       | 📋                                                       |
| Cross-browser   | Product form renders/behaves the same in Chromium, Firefox, WebKit                                   | Not covered — the suite currently runs Electron only                                                                 | 📋 — see Section 6                                       |

## 4. Cross-cutting concerns (apply across all three flows)

| Dimension                         | Observation / Recommendation                                                                                                                                                                                              | Status                                                                                                                                                                                                                                   |
| --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| **Session expiration**            | JWT lifetime is a fixed 10 minutes with no refresh flow. A logged-in user mid-form-fill for >10 minutes will have their next API call rejected with no proactive UI warning.                                              | 📋 not automatable in a fast/deterministic way — see Flow 2 rationale.                                                                                                                                                                   |
| **Concurrency**                   | The app is stateless (JWT-based), no server-side session lock. Two simultaneous "create" requests for the same unique resource (user e-mail, product name) are a real race-condition surface.                             | 📋 documented, not automated — Cypress commands are queued serially by default; true parallel requests would need raw `fetch`/`Promise.all` outside the command queue, added complexity not justified for a training API.                |
| **Network failures**              | Not exercised in this suite. `cy.intercept()` can force `forceNetworkError: true` or arbitrary delays to assert the UI's error/loading states.                                                                            | 📋 — a real, observed pain point while working on this project: the **public** ServeRest instance itself returned `503` for hours, and the front showed no fallback UI, only a stuck/blank state. A genuine finding, not a hypothetical. |
| **Timeouts**                      | No explicit request timeout handling visible in the front (`src/services/*.js` use plain `axios`/`fetch` with no `timeout` configured — confirmed by reading the source). A hung backend would hang the UI indefinitely.  | ⚠️ real gap found while reading `ServeRest/front` source.                                                                                                                                                                                |
| **Permissions / access profiles** | Two profiles only: admin (`administrador:"true"`) and regular user (`"false"`). Regular users can log in but are blocked from `/produtos` writes with a distinct `403`. Regular users **can** still read `GET /produtos`. | ✅ both write-permission boundary and public read scope are verified in `products.cy.js`.                                                                                                                                                |
| **Browser/device compatibility**  | Current suite runs Electron (headless) only. The CRA-built front (React 16, no responsive framework beyond basic CSS) was not tested at mobile viewports.                                                                 | 📋 recommend adding at least one mobile-viewport (`cy.viewport('iphone-x')`) smoke pass on `/login`.                                                                                                                                     |
| **Localization**                  | Single-locale (`pt-BR`), no currency/date formatting logic found for `preco` (raw number, no `Intl.NumberFormat`) — not a defect, just worth noting there's no i18n surface to test.                                      | 📋 documented, no action needed for this app.                                                                                                                                                                                            |
| **Accessibility**                 | Not audited. Recommend `cypress-axe` against `/login`, `/cadastrarusuarios`, `/cadastrarprodutos` as a quick automated baseline (contrast, labels, landmarks).                                                            | 📋                                                                                                                                                                                                                                       |

**Every API scenario identified in Sections 1–4 is now automated** (30 API test cases across 3
files), except the handful explicitly marked 📋 with a stated rationale (session expiration
requires real elapsed time; true concurrency needs bypassing Cypress's serial command queue). All
three core UI flows (user registration, login, product registration) now have dedicated
end-to-end coverage (7 UI test cases across 4 files), per the priorities set in Section 6.
Items marked ⚠️ are genuine gaps/inconsistencies found in the ServeRest application itself while
probing it for this analysis — not defects in this test suite.

## 6. UI automation candidate analysis

The UI layer needs a **selective** approach — automating every dimension above at the UI level
would duplicate API coverage without adding confidence, and would make the suite slow and
flaky. This section scores each remaining candidate flow by automation value.

| Candidate                                                          | User-facing risk if broken                                                       | API coverage overlap                              | Automation cost                                                                                                                    | Recommendation                                                                                                                          |
| ------------------------------------------------------------------ | -------------------------------------------------------------------------------- | ------------------------------------------------- | ---------------------------------------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------- |
| **User registration form** (happy path + inline validation errors) | High — this is the first thing a new user does; broken form = lost signups       | Field rules already proven at API level           | Low (reuses `LoginPage`-style POM, one new page object)                                                                            | ✅ Automated — `register-user.cy.js`                                                                                                    |
| **Product registration form** (admin happy path)                   | Medium — admin-only, internal tool feel                                          | Field rules already proven at API level           | Low–medium (needs login-as-admin setup + new page object)                                                                          | ✅ Automated — `product-registration.cy.js`                                                                                             |
| **Product listing / search** (`/listarprodutos`)                   | Medium — broken search blocks all shopping-cart flows downstream                 | Read endpoint already covered (`products.cy.js`)  | Low                                                                                                                                | 📋 good next candidate for a future iteration                                                                                           |
| **Duplicate-submit / button-lock on forms**                        | Low-medium — mostly a data-integrity nuisance, not a blocker                     | None (API already rejects duplicates server-side) | Medium (needs to race a real double-click, inherently a bit flaky)                                                                 | 📋 defer — server-side duplicate rejection is the real safety net                                                                       |
| **Session-expiry redirect** (protected route after token expires)  | Medium — confusing dead-end UX if it silently fails                              | None                                              | High (needs a fast way to force an expired token — e.g. seed `localStorage` with a pre-expired JWT rather than waiting 10 minutes) | 📋 worth doing with the `localStorage` trick, deferred for a future iteration                                                           |
| **Network-failure UI states** (`cy.intercept` forced errors)       | Medium — this project _lived_ this exact failure mode with the public API        | None                                              | Low (Cypress has first-class support via `cy.intercept`)                                                                           | 📋 high value given the real outage this project experienced with the public API, good candidate for a 4th UI test if the suite expands |
| **Cross-browser pass** (Firefox/WebKit in addition to Electron)    | Low for this app (no exotic browser APIs used)                                   | N/A                                               | Low (one CI matrix line)                                                                                                           | 📋 cheap to add to CI later, not a functional test per se                                                                               |
| **Accessibility baseline** (`cypress-axe`)                         | Low-medium — real users with assistive tech affected, but not a functional break | None                                              | Low (drop-in plugin + one assertion per page)                                                                                      | 📋 good "nice to have" if time remains                                                                                                  |
| **Mobile viewport smoke**                                          | Low — app is a QA-training tool, not consumer-facing at scale                    | None                                              | Very low (`cy.viewport`)                                                                                                           | 📋 cheap, low priority                                                                                                                  |

### Decision

Both **User registration** and **Product registration** score high-value/low-cost and were
automated, completing UI coverage for all three core flows (user registration, login, product
registration). **Product registration (admin happy path)** stood out as the highest-priority
addition: it's the one UI flow that (a) hadn't been touched by any existing test, (b) exercises a
distinct permission profile (admin login) end-to-end through the real UI, and (c) has zero
UI-level coverage overlap with the API suite (the API suite proves the endpoint's contract; the
UI test proves the **form** wires up to it correctly — different failure modes, both worth
covering once each). Everything else in the table above is deliberately left for a future
iteration — automating it now would trade suite speed and stability for marginal confidence.
